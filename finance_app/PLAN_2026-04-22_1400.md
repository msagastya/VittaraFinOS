## Stream 1 — SQLite Migration (Transactions + Accounts + All Controllers)

### Why it matters
Every controller today stores all data as `List<String>` (JSON-encoded rows) in SharedPreferences. Flutter's SharedPreferences is backed by a single XML file on Android that is loaded entirely into memory on first access. At ~500+ transactions the load lag becomes noticeable; at ~5,000 the file approaches the Android-enforced 8 MB SharedPreferences limit and crashes silently. There is no query capability — every filter or sort iterates the entire list in Dart. SQLite solves all of this.

### What changes

**New file: `lib/services/database_service.dart`**
- Singleton `DatabaseService` wrapping `sqflite`
- Opens/creates `vittara.db` in the app documents directory
- Runs migrations via a version counter (`_kSchemaVersion`)
- Exposes typed CRUD helpers used by all controllers

**Migration strategy (one-time, no data loss):**

Each controller, on first launch after upgrade, checks if the SQLite table is empty AND the SharedPreferences key still exists. If so, it migrates all rows from SharedPreferences → SQLite, then deletes the SharedPreferences key. After migration the app runs entirely on SQLite. If SQLite is already populated it skips the migration entirely.

```dart
// In DatabaseService
Future<void> migrateFromPrefsIfNeeded(SharedPreferences prefs) async {
  final legacyTxJson = prefs.getStringList('transactions');
  if (legacyTxJson != null && legacyTxJson.isNotEmpty) {
    final existing = await db.query('transactions', limit: 1);
    if (existing.isEmpty) {
      final batch = db.batch();
      for (final row in legacyTxJson) {
        final map = jsonDecode(row) as Map<String, dynamic>;
        batch.insert('transactions', _txToDbRow(map));
      }
      await batch.commit(noResult: true);
    }
    await prefs.remove('transactions');
  }
  // Repeat for accounts, investments, goals, budgets
}
```

| Controller | Current save | New save |
|---|---|---|

| AccountsController | `prefs.setStringList('accounts', ...)` | Same pattern |
| InvestmentsController | `prefs.setStringList('investments', ...)` | Same pattern |

**Query improvements unlocked by SQLite:**

- `searchTransactions(keyword)` — `WHERE description LIKE ? OR metadata LIKE ?`

## Stream 2 — ML Kit Entity Extraction in NL Search

### Why it matters
`_NLQueryParser` currently uses pure regex/keyword matching. It cannot understand:
- "transactions where I spent a lot on food" (no exact keyword)
- "last quarter spending" (non-standard date phrase)
- "around 2000 rupees" (approximate amounts)
- Money amounts written as "2k", "five hundred", "₹2,500"

`google_mlkit_entity_extraction` (already in pubspec, proven in VoiceFillEngine) recognizes `MoneyEntity` and `DateTimeEntity` on-device with no network call. Adding it to search gives the same power as voice parsing when typing.

### What changes

**New file: `lib/logic/search/nl_search_engine.dart`**

```dart
/// Two-pass NL search engine:
/// Pass 1 — instant rule-based (_NLQueryParser, moved here from overlay)
/// Pass 2 — ML Kit async entity extraction (~80ms)
/// Returns a stream: first item is Pass 1 result, second item (if better) is Pass 2 result
class NLSearchEngine {
  static final EntityExtractor _extractor =
      EntityExtractor(language: EntityExtractorLanguage.english);

  /// Returns Pass 1 result immediately, then emits Pass 2 result if ML Kit
  /// found additional signals (money amounts or date ranges).
  static Stream<_ParsedQuery> parseStream(String raw) async* {
    // Pass 1 — synchronous rule-based
    final p1 = _NLQueryParser.parse(raw);
    if (p1 != null) yield p1;

    // Pass 2 — ML Kit
    try {
      final annotations = await _extractor.annotateText(raw);
      _ParsedQuery? p2;
      for (final ann in annotations) {
        for (final entity in ann.entities) {
          if (entity is MoneyEntity) {
            // Enhance p1 (or create new) with ML-extracted amount
            p2 = _mergeMoneyEntity(p1, entity, raw);
          } else if (entity is DateTimeEntity) {
            p2 = _mergeDateTimeEntity(p2 ?? p1, entity);
          }
        }
      }
      if (p2 != null && p2 != p1) yield p2;
    } catch (_) {
      // ML Kit failure is non-fatal — Pass 1 result is already yielded
    }
  }
}
```

**`_GlobalSearchPageState` changes:**
- Replace `Timer` debounce → `StreamSubscription` on `NLSearchEngine.parseStream()`
- First emission updates UI instantly (Pass 1, same as today)
- Second emission (Pass 2, if different) updates results — user sees results sharpen in ~100ms
- If no second emission within 300ms, stop waiting

**Result: "2k food last month" now works even without exact keyword matching, because ML Kit extracts `MoneyEntity(2000)` and `DateTimeEntity(LAST_MONTH)` semantically.**

**Files to modify:**
- `lib/ui/widgets/global_search_overlay.dart` — wire `NLSearchEngine.parseStream()`
- `lib/logic/search/nl_search_engine.dart` — NEW (extract `_NLQueryParser` here + add ML Kit pass)

## Stream 3 — Error Handling & Data Safety

### Current state
- Controllers have `try/catch` around individual JSON row decodes but NOT around:
  - `prefs.setStringList()` saves (can fail silently if storage full)
  - The `_writeMutex` is defined in 4/5 controllers but `GoalsController` uses it inconsistently — some write paths are unprotected
  - No logging infrastructure — errors are silently swallowed or `debugPrint`-only
- Voice save in `quick_entry_sheet.dart` does NOT call `VoiceTransactionHandler` — it only pre-fills the amount/merchant fields. If the user dismisses the sheet after voice, no transaction is saved. This is a silent data loss.

### What changes

**3a. Protected save wrapper (every controller)**

All `_writeMutex.protect(...)` blocks currently have no error surface. Wrap in a `try/catch` that:
1. Logs to `debugPrint` (already exists) 
2. Sets a `lastSaveError` string on the controller
3. Calls `ToastController().showError(...)` so the user knows a save failed

```dart
// Shared mixin: lib/utils/safe_storage_mixin.dart
mixin SafeStorageMixin on ChangeNotifier {
  String? lastSaveError;

  Future<void> safeWrite(Future<void> Function() fn) async {
    try {
      await fn();
      lastSaveError = null;
    } catch (e, stack) {
      lastSaveError = e.toString();
      debugPrint('[SafeStorage] Write failed: $e\n$stack');
      ToastController().showError('Save failed — please try again');
    }
  }
}
```

**3b. GoalsController mutex consistency**

`GoalsController` currently has no `_writeMutex` at all. Add `AsyncMutex` to `GoalsController` identical to the pattern in `TransactionsController`.

**3c. Fix voice pre-fill in QuickEntrySheet**

Current code at `quick_entry_sheet.dart:1143`:
```dart
final result = await VoiceOverlayWidget.show(context);
if (result != null && context.mounted) {
  final amount = result.fields['amount'];
  // ... just pre-fills text fields, never saves
}
```

This is intentional UX (voice fills the form, user reviews then saves manually) — **this is actually correct behavior**, not a bug. The form is the review step. However, the intent field (`result.fields['_intent']`) is not applied to the sheet's `_selectedType`. Fix: after voice fill, also update `_selectedType` to match the voice intent:

```dart
final intent = result.fields['_intent'];
if (intent is VoiceIntent) {
  setState(() {
    _selectedType = intent == VoiceIntent.addIncome
        ? TransactionType.income
        : intent == VoiceIntent.addTransfer
            ? TransactionType.transfer
            : TransactionType.expense;
  });
}
```

**3d. Crash guard on SharedPreferences load**

In all controllers, the load path currently does:
```dart
final rows = _prefs.getStringList(_storageKey) ?? [];
for (final row in rows) {
  try {
    loaded.add(Model.fromMap(jsonDecode(row)));
  } catch (_) {}
}
```
This is already safe at the row level. After Stream 1 (SQLite) is implemented this entire block is replaced and this concern goes away.

## Stream 4 — AsyncMutex Consistency Audit

### Current state

| Controller | Has mutex | All saves protected |
|---|---|---|
| TransactionsController | YES (`static _writeMutex`) | YES — single `protect()` block |
| AccountsController | YES (`static _writeMutex`) | PARTIAL — balance update path in `TransactionAccountAdjuster` calls `save()` directly, which is inside mutex — OK |
| InvestmentsController | YES (`static _writeMutex`) | PARTIAL — `addActivity()` path unprotected |
| GoalsController | NO | NO |
| BudgetsController | YES (`_spendingMutex`) | PARTIAL — `onTransactionAdded` and `onTransactionDeleted` both call `_save()` but from different mutex instances |

### What changes

**Audit and fix each unprotected write path:**

1. **InvestmentsController** — `addActivity()`, `updateActivity()`, `deleteActivity()` call `_save()` without mutex. Wrap each in `_writeMutex.protect(...)`.

2. **GoalsController** — Add `static final _writeMutex = AsyncMutex()` and wrap `_save()` in every write method (`addGoal`, `updateGoal`, `deleteGoal`, `updateProgress`).

3. **BudgetsController** — `_spendingMutex` is an instance variable (not static). If two instances somehow exist (shouldn't, but Provider can recreate), they won't share a lock. Change to `static final _writeMutex = AsyncMutex()` matching other controllers.

4. **TransactionsArchiveController** (separate from transactions) — check if it has its own mutex or piggybacks on TransactionsController's. After Stream 1 this merges away, but needs guarding until then.

**Files to modify:**
- `lib/logic/investments_controller.dart` — protect `addActivity`, `updateActivity`, `deleteActivity`
- `lib/logic/goals_controller.dart` — add static mutex, protect all write paths
- `lib/logic/budgets_controller.dart` — change to static mutex
- `lib/logic/transactions_archive_controller.dart` — verify/add mutex
