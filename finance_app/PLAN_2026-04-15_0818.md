# Plan — created 15 April 2026, 08:18
## The Feel Plan — From Functional to World-Class

This is not a feature plan. This is a *feel* plan.
Every item answers one question: **"What does the user feel at this exact moment?"**

---

## The honest problem statement

The app has all the right ingredients: design tokens, spring physics, semantic colors, a font system, haptics. But they are not assembled into a coherent experience. It is like having all the instruments in a room but no conductor. The result is technically correct but emotionally flat.

World-class apps (Revolut, Apple Wallet, Monzo, Linear, Superhuman) feel the way they do because every pixel, every timing, every color relationship was decided with a specific *emotion* in mind. This plan fixes that.

---

## How this plan is structured

- **Phase** = a complete area of the experience
- **Screen** = a specific screen within that phase
- **Moment** = a specific user action or transition within that screen
- Every item specifies: what is wrong, why it feels wrong, exactly what to change

Phases run roughly in order but some are parallel. Total estimated execution: 3–4 weeks.

---

# PHASE 1 — Color Contract
## "Every color must earn its place"

This is the foundation. Nothing else works if color is wrong. Color does three jobs: it establishes hierarchy, it communicates meaning, it creates mood. Right now it does none of these reliably.

---

### 1A. Build the Elevation Language (Dark Mode)

**The problem**: In dark mode, the app is a sea of near-identical blacks. Background, cards, elevated cards, interactive surfaces — they all feel the same depth. The eye cannot tell what is tappable, what is a container, what is the base.

**The contract to enforce — exactly these values, no deviations:**

| Layer | Role | Color | Hex |
|---|---|---|---|
| L0 | App background | Pure void | `#000000` |
| L1 | Standard card / section | Dark surface | `#0D0D0D` |
| L2 | Elevated card / modal | Raised surface | `#141414` |
| L3 | Interactive surface (button bg, selected chip) | Active surface | `#1C1C1C` |
| L4 | Divider / subtle border | Hairline | `#1C1C1E` at 100% or `#FFFFFF` at 8% opacity |
| L5 | Overlay / sheet backdrop | Dimmed void | `#000000` at 60% |

**Every** `Container`, `BoxDecoration`, `CupertinoPageScaffold.backgroundColor` in the app must use one of these — no hardcoded `Color(0xFF0D1829)`, no `Colors.black.withValues(alpha:0.9)`, no inline hex that is "close enough".

**What to do**: Audit every screen, count deviations, fix to layer values. Add `AppStyles.l0`, `AppStyles.l1`, `AppStyles.l2`, `AppStyles.l3` getters that are dark-mode aware (no-ops in light mode, return their light equivalents).

---

### 1B. Build the Elevation Language (Light Mode)

**The problem**: Light mode is an afterthought. It is not designed — it is just "not dark mode." The result: cards that float with no shadow, text that blends into backgrounds, colors that are the same teal/purple as dark mode but now look saturated and jarring against white.

**The contract to enforce — exactly these values:**

| Layer | Role | Color |
|---|---|---|
| L0 | App background | Warm off-white `#F2F2F7` (iOS system grouped) |
| L1 | Standard card | Pure white `#FFFFFF` with shadow |
| L2 | Elevated card / modal | Pure white `#FFFFFF` with stronger shadow |
| L3 | Interactive surface | `#F2F2F7` (matches background — recessed feel) |
| L4 | Divider | `#C6C6C8` (iOS system separator) |

**Shadow system for light mode** (dark mode uses glow, light mode uses real shadow):
- Card shadow: `BoxShadow(color: #000000 at 6% opacity, blurRadius: 8, offset: (0, 2))`
- Elevated shadow: `BoxShadow(color: #000000 at 10% opacity, blurRadius: 16, offset: (0, 4))`
- These go into `AppStyles.cardShadow(context)` — returns glow in dark, drop shadow in light.

**What to do**: Every `AppStyles.cardDecoration()` must call `cardShadow(context)`. Remove all hardcoded `Colors.white` card backgrounds in light mode.

---

### 1C. Semantic Color Audit — Both Modes

**The problem**: `SemanticColors.info`, `.categories`, `.lending` etc. are single constant hex values. They were chosen for dark mode. In light mode, they're often too dark (low contrast on white) or too saturated (feels like a children's app).

**The fix**: Every SemanticColor must have two values — one for dark, one for light. Add a `SemanticColors.getInfo(context)`, `.getCategories(context)` etc., identical to how `AppStyles.gain(context)` and `AppStyles.loss(context)` already work.

**Specific values to define** (approximate — tune by eye after implementation):

| `warning` | `#FFD60A` | `#A05C00` |

| `categories` | `#AF52DE` | `#7B2D9E` |
| `contacts` | `#8E6B5C` | `#6B4534` |

### 1D. Text Color Contracts — Enforce Everywhere

**The problem**: Text colors are applied inconsistently. Some use `AppStyles.getTextColor()`, some use `Colors.white`, some use `Colors.white.withValues(alpha:0.55)`, some use `CupertinoColors.label` which is theme-aware but inconsistent with the design system.

**The three text roles and their exact values:**

| Role | Dark | Light | Token |
|---|---|---|---|
| Primary | `#FFFFFF` at 100% | `#000000` at 90% | `AppStyles.getTextColor()` — already exists |
| Secondary | `#FFFFFF` at 55% | `#000000` at 45% | `AppStyles.getSecondaryTextColor()` — already exists |
| Tertiary | `#FFFFFF` at 30% | `#000000` at 25% | `AppStyles.getTertiaryTextColor()` — **ADD THIS** |
| Disabled | `#FFFFFF` at 20% | `#000000` at 18% | `AppStyles.getDisabledTextColor()` — **ADD THIS** |
| On-brand (on teal/colored bg) | `#FFFFFF` | `#FFFFFF` | Always white |

**What to do**: Add `getTertiaryTextColor()` and `getDisabledTextColor()` to AppStyles. Then audit every screen for inline `Colors.white.withValues(alpha: 0.3)` or similar and replace with the correct token.

---

### 1E. Nav Bar Color Fix — Every Screen

**The problem**: Navigation bars are inconsistent. Some set `backgroundColor`, some don't. Some set `middle:` text color, some use `const Text()` with no color (breaks in light mode). Some have `border: null`, some have a visible border.

**The contract:**
- Dark nav bar background: `AppStyles.darkBackground` (`#000000`) with blur
- Light nav bar background: `Colors.white` at 95% opacity with blur
- Nav bar title: always `AppStyles.getTextColor(context)` — never hardcoded
- Nav bar border: always `AppStyles.getDividerColor(context)` at 0.5dp width, or `null` if no border is desired — never hardcoded `Colors.white.withValues()`
- Create `AppStyles.standardNavBar(context, title)` factory method that returns the correct `CupertinoNavigationBar` — then every screen calls this one method and all nav bars are immediately consistent

**Screens to fix** (screens confirmed to have nav bar issues):
- `spending_insights_screen.dart` — partially fixed in previous session, needs full audit
- `transaction_history_screen.dart`
- All `manage/` sub-screens
- `net_worth_page.dart`
- `reports_analysis_screen.dart`
- Every screen under `manage/loans/`, `manage/insurance/`, `manage/fd/`, `manage/stocks/`

# PHASE 2 — Typography as Hierarchy
## "The eye must know where to go without being told"

---

### 2A. Establish a Typography Contract Per Screen Type

Every screen in the app falls into one of 4 types. Each type has a fixed typography spec that cannot deviate:

**Type A — Dashboard / Hero Screens**
- Screen title: `AppTypography.largeTitle()` — size 34, weight 700, letterSpacing -0.5
- Section label: `AppTypography.caption()` — size 11, weight 700, ALL CAPS, letterSpacing 1.2, secondary color
- Card title: `AppTypography.headline()` — size 17, weight 600
- Card value (money): `AppTypography.currencyLarge()` with `FontFeature.tabularFigures()`
- Card subtitle: `AppTypography.footnote()` — size 13, weight 400, secondary color

**Type B — List Screens (Accounts, Transactions, Banks)**
- Nav bar title: `AppTypography.title2()` — size 22, weight 700 (Space Grotesk)
- Section header: `AppTypography.caption()` ALL CAPS + secondary color
- List item title: `AppTypography.callout()` — size 16, weight 500
- List item detail: `AppTypography.footnote()` — size 13, weight 400, secondary color
- List item value: `AppTypography.callout()` with tabular figures, right-aligned

**Type C — Form / Wizard Screens**
- Step title: `AppTypography.title2()` — size 22, weight 700
- Field label: `AppTypography.caption()` — size 11, ALL CAPS, secondary color
- Field value / input: `AppTypography.body()` — size 17, weight 400
- Helper text: `AppTypography.footnote()` — secondary color
- CTA button: `AppTypography.button()` — size 16, weight 700

**Type D — Detail Screens (Investment detail, Loan detail)**
- Hero number: `AppTypography.hero()` or `currencyLarge()` — size 34–40
- Hero label: `AppTypography.caption()` under the hero number
- Stats grid label: `AppTypography.micro()` or `caption()` ALL CAPS
- Stats grid value: `AppTypography.callout()` with tabular figures

---

### 2B. Remove Every Inline TextStyle

**The problem**: The codebase has hundreds of inline `TextStyle(fontSize: 14, fontWeight: FontWeight.w600, color: ...)` definitions scattered throughout. These bypass the typography system entirely. Two cards on the same screen can have the same "title" text but different sizes, weights, and letter spacings because both were written inline on different days.

**The rule**: No file may define a `TextStyle(fontSize: ...)` inline unless it is inside `AppTypography`, `typography.dart`, or `app_styles.dart`. Every other occurrence is a violation.

**Method**: Grep every screen file for `TextStyle(fontSize` — replace with the correct `AppTypography.*` call.

This is mechanical work but it is the most important change in this plan. Typography consistency is what separates an app that "looks designed" from one that "looks developed."

---

### 2C. Number Formatting — Tabular Figures Everywhere

**The problem**: Currency amounts and numbers use proportional figures by default. When a number changes (e.g., balance updates from `₹12,450` to `₹1,450`) the surrounding text shifts because digits have different widths. This makes the UI jitter.

**The fix**: Every number that can change, every currency amount, every percentage — must use `FontFeature.tabularFigures()`. This is already in `AppTypography.numeric()`, `currencyLarge()`, `currencyMedium()`, `currencySmall()`. The fix is ensuring every number display uses these methods.

# PHASE 3 — Spacing as Breathing Room
## "Space is not empty — it is the pause that gives meaning"

---

### 3A. Remove All Hardcoded Spacing

**The problem**: `EdgeInsets.all(16)`, `SizedBox(height: 8)`, `padding: const EdgeInsets.fromLTRB(20, 12, 20, 8)` — these are everywhere. Each was typed by hand on a different day. The result: the app has no consistent spacing rhythm. Some cards have 16dp padding, others 20dp. Some list items have 12dp vertical spacing, others 8dp.

**The token system to enforce** (already in `design_tokens.dart`):

| Token | Value | Use for |
|---|---|---|
| `Spacing.xxs` | 2dp | Tick marks, minimal gaps |
| `Spacing.xs` | 4dp | Icon-to-label gap, tight list spacing |
| `Spacing.sm` | 8dp | Inner card spacing, between related elements |
| `Spacing.md` | 12dp | Standard inner padding |
| `Spacing.lg` | 16dp | Card padding horizontal, section gaps |
| `Spacing.xl` | 20dp | Large section separation |
| `Spacing.xxl` | 24dp | Hero section padding |
| `Spacing.xxxl` | 32dp | Between major sections |
| `Spacing.massive` | 64dp | Bottom safe area padding |

**The rule**: No file may use `EdgeInsets.all(N)` or `SizedBox(height: N)` where N is a raw number. Every spacing value must be a `Spacing.*` token. No exceptions.

---

### 3B. Card Padding Standardization

Every card in the app must use one of these padding specs — nothing else:

| Card type | Padding |
|---|---|
| Standard list card | `EdgeInsets.symmetric(horizontal: Spacing.lg, vertical: Spacing.md)` |
| Hero/dashboard card | `EdgeInsets.all(Spacing.xl)` |
| Compact row card | `EdgeInsets.symmetric(horizontal: Spacing.lg, vertical: Spacing.sm)` |
| Bottom sheet inner | `EdgeInsets.fromLTRB(Spacing.lg, Spacing.sm, Spacing.lg, Spacing.xl)` |
| Section within sheet | `EdgeInsets.symmetric(horizontal: Spacing.lg, vertical: Spacing.md)` |

---

### 3C. Minimum Touch Target Audit

**The problem**: Some tappable elements in the app are smaller than 44×44dp (Apple's minimum). Small tap targets make the app feel cheap and frustrating.

**Every tappable element must be at minimum:**
- Height: 44dp
- Width: 44dp
- If the visible widget is smaller (e.g., a 20dp icon), wrap it in a `GestureDetector` with `behavior: HitTestBehavior.opaque` and explicit size constraints

**Screens to audit specifically:**
- Quick entry sheet number pad buttons
- Transaction history filter chips
- Manage screen reorder handles
- Settings toggle rows
- Nav bar back buttons

# PHASE 4 — The Dashboard Flow
## "The first 3 seconds must tell the user they are in control"

The dashboard is the most important screen. It is seen every single time the app opens. It sets the emotional tone for everything that follows.

---

### 4A. First-Launch Dashboard Experience

**Current state**: A new user completes onboarding, arrives at dashboard, sees an empty card deck (or the setup card I built). There is no guidance, no warmth, no sense of possibility.

**What it should feel like**: Like turning on a powerful machine for the first time. Lights come on. Systems initialize. It feels capable.

**The specific changes:**
1. On first launch (accounts empty), the setup card entrance animation must be a stagger — the icon appears first (spring scale from 0.5→1.0 with `AppSprings.bouncy`), then the title slides up (120ms delay), then the subtitle (200ms delay), then the button (300ms delay). Not all at once.
2. The header section must also animate in — greeting fades+slides up, date subtitle follows 80ms later, quick action pills stagger in with 40ms gaps.
3. The FAB should pulse once (subtle scale 1.0→1.06→1.0 with `AppSprings.gentle`) after 800ms on first launch, drawing the eye to "this is how you add things."

---

### 4B. Daily Dashboard — What the Eye Sees First

**Current state**: The header has a greeting, a date, the "Net Worth" badge, and quick action pills. The card deck starts below. The user's most important number — their total net worth or total balance — is buried in one of the deck cards.

**What world-class apps do**: They put the *single most important number* at the very top. Apple Wallet: your card balance. Revolut: your total balance. Trading 212: your portfolio value. The number is the hero. Everything else is context.

**The change**: The header section should display the current net worth / total account balance as a hero number directly visible without swiping. Large `AppTypography.currencyLarge()`, tabular figures, animates with `AnimatedCounter` scramble on load. Secondary label underneath says "Total across N accounts" in secondary color caption size.

**Specific layout:**
```
[Good Morning          ] [Net Worth →]
[Wednesday, 15 Apr     ]
                         
  ₹4,82,350              ← hero number, size 34, weight 700
  Across 3 accounts      ← caption, secondary color

[History] [Budgets] [Goals] [Savings] [AI Plan]   ← pills
```

---

### 4C. Dashboard Card Deck — Discoverability

**Current state**: The card deck shows 3 depth levels. Users do not know they can swipe because nothing on the first open suggests it.

**The fix — exactly:**
1. On dashboard **first open** (check SharedPreferences `dashboard_swipe_hint_shown`): the top card should auto-slide 15dp to the right and back within 600ms on a `SpringSimulation(AppSprings.gentle, 0, 15, 0)` then `SpringSimulation(AppSprings.natural, 15, 0, -3)`. This is the universal "swipeable" hint.
2. After this happens once, store `dashboard_swipe_hint_shown = true` and never do it again.
3. The depth peek (already implemented) makes subsequent swipeability clear.

---

### 4D. Dashboard Card Drag Handle — Remove It

**Current state**: Each dashboard widget card has `CupertinoIcons.line_horizontal_3` (the hamburger menu / drag handle icon). This implies the card is draggable / reorderable. It is not — swiping cycles the deck. The icon lies to the user.

**The fix**: Remove the three-line icon entirely from dashboard widget cards. If the card needs an action, use `CupertinoIcons.ellipsis` or nothing at all.

---

### 4E. Card Deck — Bottom Dot Indicators

**Current state**: Pills of varying width animate between cards. They work, but they sit at the bottom of the screen below the card, disconnected from the card content.

**Improvement**: The dots should be inside the card deck area, overlaid at the bottom of the card like a carousel. Positioned absolutely inside the `Stack`, 12dp from the bottom of the card. This connects them visually to the card they represent.

# PHASE 5 — Transaction Entry Flow
## "This is the most frequent action. It must be perfect."

Every time someone spends money, they open this. On average, 10-20 times per week. If it feels even slightly rough, it accumulates into frustration.

---

### 5A. Quick Entry Sheet — Open Animation

**Current state**: The sheet slides up from bottom with standard `showCupertinoModalPopup`. The content appears fully formed, instantly.

**What it should feel like**: The sheet arrives with energy. The keyboard feels ready. The amount field feels like it's waiting for you.

**The fix:**
1. On sheet open, the amount display (the big number area) should scale from 0.95→1.0 with `AppSprings.natural`, starting 120ms after the sheet arrives.
2. The amount `0` should flash a cursor blink once — just a 200ms opacity pulse on the cursor indicator.
3. The type pills (Expense/Income/Transfer) should stagger in left-to-right with 40ms gaps.

---

### 5B. Quick Entry Sheet — Number Entry Feedback

**Current state**: Tapping number buttons adds to the amount. The feedback is visual only.

**Required haptic contract:**
- Digit tap: `Haptics.light()` — already there but confirm it's on EVERY digit button
- Decimal tap: `Haptics.light()`
- Backspace tap: `Haptics.selection()`
- Clear (long press backspace): `Haptics.medium()`
- Amount change between expense/income/transfer: `Haptics.selection()`
- Save button: `Haptics.success()` — the double-tap pattern, feels final

---

### 5C. Quick Entry Sheet — Amount Display

**Current state**: The amount shows as text with a standard cursor.

**The improvement**: When the amount is `0` or empty, the text should pulse softly at 20% opacity → 60% opacity → 20% opacity with a 1.5s period, indicating "waiting for input." The moment any digit is entered, the pulse stops. This is a common pattern in payment apps — it signals readiness without text.

---

### 5D. Quick Entry Sheet — Save Confirmation

**Current state**: Taps save → toast appears → sheet dismisses. The user sees nothing that says "this worked." They have to read the toast.

**The improvement**: Before the sheet dismisses, show a full-sheet success flash:
- Sheet background flashes to `SemanticColors.success.withValues(alpha: 0.12)` for 200ms
- The save button morphs: background fills with solid `SemanticColors.success`, icon changes to checkmark, scale bounces with `AppSprings.bouncy`
- This lasts 350ms, then sheet dismisses
- Total added time: ~500ms — imperceptible, but the user *feels* the confirmation
- Simultaneously: FAB checkmark morph (already built)

# PHASE 10 — Motion Choreography
## "Every transition is a sentence. It must have grammar."

---

### 10A. The Transition Grammar

Define exactly three transition types and use each consistently:

**T1 — Push (go deeper into hierarchy):**
- New screen arrives: scale 0.94→1.0 spring, fade 0→1 linear 200ms
- Old screen exits: scale 1.0→0.96, fade 1→0.6 (not fully transparent — stays present)
- Feel: diving forward

**T2 — Modal (sheet / overlay):**
- Sheet arrives: slide up from bottom + background dims to 60% black
- Sheet dismisses: slide down, background brightens back
- Duration: 320ms spring with `AppSprings.gentle`

**T3 — Replace (tab-like, same hierarchy level):**
- Outgoing: fade to 0 + scale to 0.98
- Incoming: fade from 0 + scale from 0.98
- Duration: 200ms total, crossfade
- Used for: switching between card deck cards, switching between tabs in a settings section

**Currently T1 is partially implemented (`FadeScalePageRoute`). T2 and T3 need implementation.**

---

### 10B. Loading → Content Transition

**The rule**: No screen may show content without first showing a skeleton loader. The skeleton must be visible for at minimum 200ms (even if data loads instantly) — this prevents jarring flash.

**Implementation**: Every `Consumer` / `Selector` that builds a list or grid must:
1. Check `controller.isInitialized` — if false, show `SkeletonAnimationProvider` wrapping a column of `SkeletonLoader.listItem()` × 5
2. When data arrives, crossfade from skeleton to content using `AnimatedCrossFade` with 200ms duration

---

### 10C. Error State Animations

**Current state**: Errors appear as toast notifications. The tapped widget does nothing visual when an error occurs.

**The add**: Any form field with validation error should shake (horizontal oscillation, `ShakeAnimation` already exists). Any save button that fails should pulse red — background flashes to `SemanticColors.error.withValues(alpha: 0.3)` for 200ms. Haptic: `Haptics.error()`.

# PHASE 11 — Light Mode as a First-Class Experience
## "Light mode is not dark mode with inverted colors. It is a different emotional register."

This phase is about treating light mode as a designed experience, not a fallback.

---

### 11A. Light Mode Mood Board

The AETHER dark mode has a specific mood: bioluminescent, deep space, emissive. Light mode must have its own mood: **crisp, clean, trustworthy**. Like a well-lit bank. Professional but approachable.

**Light mode color philosophy:**
- Backgrounds: warm off-white (not cold white) — `#F2F2F7`
- Cards: pure white with subtle drop shadows (elevation through shadow, not lightness)
- Accent: deeper, more muted versions of the brand teal — `#007A62` not `#00B890`
- Success/gain: deep forest green — `#248A3D` not `#34C759`
- Error/loss: deep crimson — `#C00B0B` not `#FF453A`
- The palette leans slightly warm and muted — the opposite of dark mode's vivid emission

---

### 11B. Dashboard Header in Light Mode

**Current state**: The dashboard header card has ambient glow orbs and a dot grid. In dark mode: beautiful, atmospheric. In light mode: the glow orbs become washed-out teal blobs on a pale card. The dot grid disappears entirely (dots are white on white). The effect that made it feel special in dark mode just looks broken in light mode.

**The fix**: The header card must have two distinct visual treatments:
- **Dark mode**: emissive orbs + dot grid (current)
- **Light mode**: a clean `LinearGradient(colors: [white, #F0FAFA])` — subtle teal tint at the bottom. No orbs. No grid. Just clean gradient with the text. The elegance is in the restraint.

---

### 11C. Icon Boxes in Light Mode

**Current state**: `AppStyles.iconBoxDecoration(context, color)` creates an icon box. In dark mode: a dark background with a subtle color tint. In light mode: this may produce a washed-out pastel that lacks identity.

**The fix**: In light mode, icon boxes should use `color.withValues(alpha: 0.12)` background + a 1dp border at `color.withValues(alpha: 0.25)`. This creates a soft, branded feel that looks designed on white backgrounds. In dark mode: keep current (`color.withValues(alpha: 0.18)` background, no border).

# PHASE 12 — Small Details That Make the Difference
## "The last 10% is what separates good from unforgettable"

---

### 12A. The Modal Handle

**Current state**: `ModalHandle` widget exists and is used in most sheets. Size: `width: 36, height: 4`. Token: `ComponentSizes.modalHandleWidth = 40, height = 5`.

**Fix**: Use the token. Also: the handle color in dark mode should be `Colors.white.withValues(alpha: 0.3)`. In light mode: `Colors.black.withValues(alpha: 0.2)`. Currently it may be a hardcoded value.

---

### 12B. Empty State Design — All Screens

Every screen that can show an empty list must show a designed empty state. Not a blank area, not just text. Each empty state has:
1. An icon (relevant to the screen, size 48dp, `SemanticColors.getForScreen(context)`)
2. A title: `AppTypography.headline()` primary color, centered
3. A subtitle: `AppTypography.callout()` secondary color, centered, max 2 lines
4. An action button (CTA): `AppButton.primary()` — not `CupertinoButton`

**Screens that currently have blank or missing empty states:**
- Transactions (no transactions yet)
- Goals (no goals)
- Budgets (no budgets)
- Insurance (no policies)
- Contacts (no people)
- Tags (no tags)

---

### 12C. Pull-to-Refresh Indicator

**Current state**: Standard `CupertinoSliverRefreshControl` — the iOS spinner. This is fine functionally but has no brand personality.

**The minimum fix**: On `onRefresh` trigger, `Haptics.medium()`. The indicator color should use `AppStyles.getPrimaryColor(context)` — currently it defaults to iOS blue.

---

### 12D. FAB Visibility on Scroll

**Current state**: The FAB is always visible. When the user scrolls down in a list, the FAB overlaps content.

**The fix**: FAB should fade out when scrolling down (fade + scale 1.0→0.8 with `AppSprings.snappy`) and reappear when scrolling stops or reverses. This already partially exists as `FadingFAB` in `common_widgets.dart` — wire it in where needed.

---

### 12E. Keyboard Avoidance

**Current state**: When a keyboard appears in form screens, content may be obscured behind it.

**The fix**: Every form screen that has text input must use `resizeToAvoidBottomInset: true` on its scaffold. If a `ListView` is the scrollable container, it should have `keyboardDismissBehavior: ScrollViewKeyboardDismissBehavior.onDrag` — so scrolling the list dismisses the keyboard naturally.
