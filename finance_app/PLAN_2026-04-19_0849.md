# Landscape Mode — Full Redesign

---

## Why the current approach fails

The app has ~55 screens. Every single one just:
1. Hides the `CupertinoNavigationBar`
2. Adds a 40-44px compact inline bar at the top
3. Lets the portrait column stretch to full width

This is not a landscape experience. It is a portrait experience on a wide screen. The result:
- Content is spread out horizontally but still scrolls vertically in a thin centre strip
- The extra width buys nothing — you still scroll as much as in portrait
- On short landscape screens (~360px tall) you have almost no visible content at once

The right mental model: **landscape is not portrait rotated. It is a completely different viewport** — wide + short. The design must use width to eliminate height.

---

## Core design principles

### Principle 1: Every landscape screen is a horizontal composition
- Content is arranged **left → right**, not top → bottom
- Left panel = navigation, context, summary (static / pinned)
- Right panel = primary scrollable content

### Principle 2: Left panel serves a different purpose per screen type
| Screen type | Left panel role | Left width |
|---|---|---|
| Hub / menu | Vertical nav menu | 200–220px |
| List + detail | List / master view | 45% |
| Summary + list | Summary / stats card | 200–240px |
| Chart + table | Chart or key metrics | 45% |
| Wizard / form | Steps + context card | 220–240px |
| Calendar | Month grid | 45% |

### Principle 3: No stacked headers in landscape
In portrait: `[header] [list]` stacked vertically is fine.  
In landscape: the header goes into the left panel. The list takes 100% of the right panel height.  
This way the list has full screen height (minus nav bar) on the right.

### Principle 4: Navigation bars become left rails
Every screen that currently has a top bar in landscape gets a left rail instead:
- Back button at the very top of the rail
- Screen title below it
- Context info (count, date, amount) in the rail
- Action buttons (filter, sort, export) at the bottom of the rail

### Principle 5: Compact information density
In landscape, text and spacing scales down slightly:
- Section labels: 9px caps
- Body: 12px
- Rail items: 34px tap targets (vs 44px portrait)
- Padding: Spacing.sm (8px) instead of Spacing.lg (16px) in rails

### Principle 6: Wizards use left-step-rail + right-form
All 15+ wizard screens get the same treatment:
- Left (220px): step indicator (vertical, numbered) + context summary card
- Right: the form content, full height, no need to compress

## Screen type taxonomy & landscape pattern

### TYPE A — Hub Screens (ManageScreen, AppMenuScreen)
**Current**: Single column list  
**Landscape**: 2-column or 3-column card grid, full height  
Pattern: `[compact nav bar 44px][grid of category cards]`  
No split needed — grid uses full width naturally

### TYPE B — Summary + Scrollable List
Screens: GoalsScreen, BudgetsScreen, LendingBorrowingScreen, NotificationsPage, TransactionsArchiveScreen, InsuranceScreen  
**Current**: Summary card stacked above list, both scroll  
**Landscape**: Summary card pinned LEFT (220px), list scrolls RIGHT  
- Left: back button, screen title, summary card (can be taller now — no scroll needed), action buttons at bottom
- Right: filter chips row at top, then full-height scrollable list

### TYPE C — Tab + List
Screens: AccountsScreen, InvestmentsScreen, NotificationsPage  
**Current**: Horizontal tab chips scroll, PageView content below  
**Landscape**: Vertical tab pills on LEFT (180px), tab content fills RIGHT  
- Left: back, title, vertical tab list (each tab = 40px row)
- Right: content of selected tab, full height

### TYPE D — Calendar + Events
Screen: FinancialCalendarScreen  
**Landscape**: Month calendar pinned LEFT (45%), event list scrolls RIGHT  
- Left: compact back+title bar, then the month grid
- Right: upcoming strip + event list

### TYPE E — Analytics / Charts
Screens: SpendingInsightsScreen, ReportsAnalysisScreen, TaxSummaryScreen, BudgetDetailsScreen  
**Current**: Charts stacked above tables, all scroll  
**Landscape**: Key metrics + chart controls LEFT (38%), scrollable charts+tables RIGHT  
- Left: back, title, key stats pills (spent/income/saved), date range selector
- Right: all the chart content, scrollable

### TYPE F — Card Deck
Screens: DashboardScreen, NetWorthPage  
**Dashboard landscape**: Already redesigned (sidebar + deck) — refine sizing  
**NetWorthPage landscape**: Left (45%) = vault summary + net worth ring, Right (55%) = 2×2 mini cards for each category  

### TYPE G — Transaction History
Already redesigned in the previous iteration — but the left panel needs to be richer:
- Left: back, title, search, date range quick-select (Today/Week/Month/Year as vertical chips), stats, filter toggle
- Right: grouped transaction list

### TYPE H — Settings
Screen: SettingsScreen  
**Landscape**: Settings category list LEFT (220px), selected category content RIGHT  
- Left: back, SETTINGS label, vertical list of setting sections (Privacy, Display, Data, About, Danger Zone)
- Right: the settings for the tapped section

### TYPE I — Wizard / Form
All 15+ wizard screens  
**Landscape**: Step indicator column LEFT (220px), form content RIGHT  
- Left: back button, wizard title, vertical numbered step list with current step highlighted, context summary (what has been filled so far)
- Right: current step form, full height, next/back at bottom

### TYPE J — Detail screens (investment details, loan details, etc.)
These are full-screen detail views  
**Landscape**: Key stats panel LEFT (40%), scrollable detail content RIGHT  
- Left: back, asset name + type, key number (current value, P&L, rate), a compact sparkline if available
- Right: all the detail content (history, schedule, breakdown)

## Reusable landscape infrastructure to build

Before touching any screen, build these 3 shared components:

### 1. `LandscapeRail` widget
A reusable left-panel container used by all screens.

```
LandscapeRail({
  required Widget child,        // the rail content
  double width = 210,
  bool showBorder = true,       // right border divider
})
```

Internally: Container with fixed width, background = cardColor at 0.95 alpha, right border = dividerColor 0.5px.

### 2. `LandscapeRailHeader` widget
The back button + title bar used at the top of every rail.

```
LandscapeRailHeader({
  required String title,        // e.g. 'GOALS'
  required BuildContext context,
  Widget? trailing,             // optional action icon
  VoidCallback? onBack,         // defaults to Navigator.maybePop
})
```

Renders a 44px row: [chevron_left back button] [title text 11px caps] [spacer] [trailing?]

### 3. `LandscapeScaffold` widget
Wrap any screen body to get the two-panel layout automatically.

```
LandscapeScaffold({
  required Widget leftRail,
  required Widget body,
  double railWidth = 210,
  bool portrait: full-width body,
})
```

In portrait: returns `body` only (full width).
In landscape: returns `Row([LandscapeRail(child: leftRail), Expanded(child: body)])`.

These 3 components eliminate all the duplicated landscape nav bar code (~10 copies).
